第一段 · 00:00

先說四個真實案例

這些不是假設——是判決、是訴訟、是一夜變天的授權。
每一個都藏著金融 IT 最容易踩的陷阱。

案例一 · 開源授權「告得贏」

Artifex v. Hancom（2017，美國加州北區法院）

發生什麼

Ghostscript 雙重授權被踩線

Ghostscript 採「AGPL 或付費商業授權」雙軌。Hancom 在其 Office 產品中使用，既不付商業授權、也不遵守開源義務。Artifex 提告。

法院怎麼判

開源授權＝有效契約

法院認定：違反 GPL/AGPL 可同時構成著作權侵權與「違約」，原告得求償。確立開源授權在法庭上「站得住、告得贏」。

  • 陷阱：「免費版＝可商用」是錯覺 — 雙重授權的免費那軌往往是 AGPL
案例二 · 金融報表的隱藏地雷

iText PDF：你的對帳單可能正踩著它

為何貼近金融

PDF 報表 / 對帳單 / 表單產生

iText 是極普及的 PDF 函式庫，金融業大量用於產製報表、對帳單、KYC 表單。其開源版本是 AGPL，商用須另購授權。

典型誤用

工程師「找個套件就用」

開發者依套件名搜尋、直接引入，未察覺授權。一旦對外提供服務，整套報表系統落入 AGPL 揭露範圍，事後常需補繳高額商業授權費。

  • 陷阱：「常用＝安全」是錯覺 — 越普及的元件，越可能是 copyleft 變現的標的
案例三 · AI 補全的原罪

GitHub Copilot 集體訴訟（Doe v. GitHub, 2022）

爭點

AI 吐出受保護碼、移除授權標示

原告主張 Copilot 以大量開源碼訓練，會「逐字」產出他人程式碼卻不附帶授權與出處，涉嫌移除著作權管理資訊（DMCA §1202）。

對你的意義

污染從「選型」變成「逐行」

就算你沒主動引入任何 AGPL 套件，AI 補全也可能把無標示的 copyleft 片段帶進核心。傳統 license 掃描看不到——因為沒有 LICENSE 檔可比對。

  • 陷阱：「AI 寫的就沒事」是錯覺 — 責任仍在公司，不能用「AI 產的」免責
案例四 · 你的依賴一夜變天

MongoDB · Elastic · Grafana：授權變更潮

2018

MongoDB → SSPL

由 AGPL 再轉更嚴格的 SSPL；Debian / RHEL 一度下架，商用者被迫重新評估。

2021

Elastic → SSPL

反制雲端業者託管變現，引發 AWS 另立 OpenSearch 分支。

2021

Grafana → AGPLv3

由寬鬆的 Apache-2.0 改為 AGPLv3，既有使用者的合規前提瞬間改變。

  • 陷阱：「當初是寬鬆授權」是錯覺 — 授權是會變的商業武器，升級／鎖版前必須重檢
開場收束

四個案例，四個共通陷阱

🎁

免費版 ≠ 可商用

雙重授權的免費軌常是 AGPL（Ghostscript、iText…）。

📦

常用 ≠ 安全

越普及的元件越可能成為 copyleft 變現的標的。

🤖

AI 產 ≠ 免責

補全可能植入無標示的 copyleft 片段，責任仍在你。

🔄

當初寬鬆 ≠ 永遠寬鬆

授權會變更，依賴管理必須持續重檢。

接下來：把這四個陷阱放回「授權地圖」，你就知道該怕什麼、怕到什麼程度。

第二段 · 00:15

授權光譜與 copyleft 基本盤

不用背法條，先建立一個直覺：授權像一條光譜，越往右、對金融伺服器服務的殺傷力越大。

一張圖看懂相對風險

從「拿去用就好」到「連你的系統一起公開」

MIT / BSD寬鬆
Apache-2.0寬鬆＋專利
LGPL / MPL弱傳染
GPL-3.0強傳染·散布觸發
AGPL / SSPL網路即觸發

寬鬆（左）

可商用、可閉源，僅需保留聲明。金融系統首選。

弱傳染（中）

改動該元件才需回饋，通常不感染你的主程式。可控。

強傳染（右）

構成衍生作品即須揭露原始碼；AGPL 連「網路服務」都觸發。

第三段 · 00:30

AGPL §13
網路互動條款

GPL 與 AGPL 只差一條——但這一條，正好打中金融業的命門。

唯一的致命差異

GPL 防「拿走」，AGPL 防「躲在伺服器後面拿走」

GPL-3.0 §0

伺服器執行 ≠ 散布

把軟體放在伺服器、只把「結果」回給使用者，不算散布，不需提供原始碼。這就是「SaaS / ASP 漏洞」。

AGPL-3.0 §13

網路互動 ＝ 視同散布

使用者透過網路遠端互動，即視同散布，必須對每一位使用者提供完整對應原始碼（含你的修改）。漏洞補上。

為什麼金融首當其衝

你最值錢的資產，剛好躲在伺服器後面

  • 金融系統幾乎都是伺服器端 / 雲端服務 — 網銀、App、開放 API，正是 §13 的射程
  • 核心是專有的交易邏輯、風控模型、定價演算法 — 一旦與 AGPL 元件構成同一作品，理論上全落入揭露範圍
  • 還疊著監理與個資義務 — 揭露原始碼可能同時牴觸資安與保密要求

⚖️ 對一般 SaaS，AGPL 是「需謹慎」；對金融機構，是 「預設禁用、例外才放行」 的等級。

AI 時代的開源應用
Claude · Copilot · RAG · MCP

生成式 AI 重寫了「開源風險」的形狀——它同時是最大的新破口，也是最強的新防線。

AI 開發的新常態

今天一個金融 AI 功能的 stack，長這樣

🧠

基礎模型

Claude 等以 API / 雲端服務呼叫；多數開源框架（LangChain、vLLM）為寬鬆授權。

🔎

RAG / 向量庫

檢索增強生成；須逐一確認向量資料庫與工具鏈授權，勿假設都寬鬆。

🔌

MCP / 工具伺服器

大量社群開源 MCP server，授權品質參差，引入前須查 LICENSE。

✍️

AI 編碼助手

Claude、Copilot 直接產出進 codebase 的程式碼——風險的新入口。

關鍵心法：「用 AI 服務」與「散布 AI 產出的程式碼」是兩件事。前者通常無 copyleft 問題；後者才是污染來源。

新型污染鏈

風險怎麼從 AI 補全溜進核心

🤖

AI 補全 / 貼上

產出看似原創、實為複製的片段

→
🫥

無授權標頭

沒有 SPDX / LICENSE，metadata 掃描失效

🏦

上線、對外互動 → 隱性觸發 §13

為什麼特別難防

無 metadata → 傳統授權掃描看不到，必須靠程式碼相似度（snippet-level）比對；潛伏期長，常在稽核或併購 due diligence 才爆。

第一道對策

AI 工具開啟公開程式碼過濾 / 來源標示；PR 模板加入「來源宣告」欄位，建立可追溯紀錄。

雙面刃

反過來——用 AI 來防 AI

🧭

授權偵測助手

讓 AI 對可疑片段做相似度比對、辨識來源專案與授權，補 SCA 工具死角。

📋

SBOM / 盤點生成

用 AI 加速整理元件清單、授權矩陣與風險摘要，產出可審查的 SBOM。

🛡️

PR 授權守門

在 code review 階段，讓 AI 標記「這段像某 AGPL 專案」，把問題擋在合併前。

原則：AI 是放大器。沒有政策與流程，它放大風險；有了治理，它放大你的防禦力。

第五段 · 01:05 · 互動

輪到你——定位自己的風險

打開手機掃 QR，到互動網站點選你的系統落在熱力圖哪一格、跑一次決策樹。

同元件，不同用法，風險天差地別

金融場景 × 整合方式

低風險

你只是它的網路客戶端

未修改、未散布，呼叫其服務 → 通常不觸發。仍須留存證據。

中風險

獨立行程 · IPC / CLI

耦合較鬆，是否構成衍生作品為灰色地帶，須法務就架構認定並做隔離。

極高風險

靜態連結 / 同行程嵌入

緊耦合＋對外服務 → 幾乎必然構成衍生作品並觸發揭露。立即禁用或購商業授權。

越緊密耦合、越對外提供網路互動，風險越往紅色端移動。

30 秒自評

「這個元件我能用嗎？」

  • ① 它是 AGPL / SSPL（或不確定）嗎？ 否、是 MIT/Apache → 低風險，留存 SBOM 即可
  • ② 會與系統整合並對網路使用者服務嗎？ 只在內部離線 → 中風險，建立隔離與標記
  • ③ 整合方式是緊耦合（靜態連結 / 同行程 / 改其碼）嗎？ 是 → 極高風險，停用或購商業授權
  • ④ 鬆耦合但無法替換 / 無商業授權？ → 灰色地帶，上線前必須取得法務書面結論

互動版在學習網站，可現場分組各跑一條路徑、上台報告判讀。

第六段 · 01:20

防範縱深與行動清單

防 AGPL 不是單一工具，是一條從「政策 → 工具 → 流程 → 法務」的縱深防線。

六道防線

從政策到法務，一層都不能少

📜

1. 授權政策

白／黑名單，明定 AGPL/SSPL 禁用或須例外審批。

🔬

2. 片段級 SCA

導入掃描並開啟 snippet 相似度比對，非僅套件層級。

🚦

3. CI/CD 門禁

偵測到 AGPL 即 fail build，擋在合併前。

🧾

4. SBOM 可追溯

每次發布產出元件與授權清單，稽核／併購可交代。

🤖

5. AI 工具治理

開啟公開碼過濾、PR 來源宣告，納入新型風險。

⚖️

6. 例外＋法務簽核

灰色地帶一律送審，書面結論前不得上線。
